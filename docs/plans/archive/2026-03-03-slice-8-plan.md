# Opal Slice 8: Web Server Capstone with FFI

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add a plugin-based FFI system (`extern "name"` blocks), build an HTTP server plugin using `std::net::TcpListener`, and deliver a working web server with `@get` macro routing.

**Architecture:** `extern "name"` declares foreign functions resolved from a `PluginRegistry`. The `http` plugin provides `create_app`, `add_route`, `run` as native functions. Route handlers are Opal closures called back from Rust during the request loop. `@get` is a macro that desugars to `add_route`. A new `Value::NativeObject(NativeObjectId)` holds opaque Rust-side state (the router).

**Tech Stack:** Rust `std::net::TcpListener` for HTTP. No new external dependencies.

**Design doc:** `docs/plans/2026-03-03-slice-8-design.md`

---

## Context

This is the Phase 1 capstone. After this slice, Opal has: strings, numbers, functions, closures, lists, classes, modules, actors, macros, error handling, pattern matching, FFI, and a working web server.

**The handler callback challenge:** The HTTP accept loop runs in Rust, but handlers are Opal closures. Solution: the `run` native function receives the interpreter (via a trait object callback) and uses it to invoke closures. The interpreter's `call_closure` method is already public-compatible — we just need to expose it through a callback interface.

### Task 4: Implement extern handling in interpreter

**Files:**
- Modify: `crates/opal-interp/src/eval.rs`
- Modify: `crates/opal-interp/Cargo.toml` (add opal-lexer dependency for Span, if needed)

**Add `plugin_registry: PluginRegistry` and `native_objects: Vec<Box<dyn Any>>` to Interpreter.**

**Handle `StmtKind::ExternDef`:** Look up plugin in registry, bind each declared function name to a `Value::Function` (or new `Value::NativeFunction`) that wraps the plugin's native function.

Actually, the simplest approach: introduce `Value::NativeFunction(NativeFunctionId)` — an ID into a table of registered native function references. When called, the interpreter dispatches to the native function.

Alternatively, store extern functions in the environment as a special callable. When `eval_call` encounters an identifier, check if it resolves to a native function and dispatch.

**The callback problem:** `http_run` needs to call Opal closures during the HTTP loop. The cleanest solution: the `run` native function doesn't directly run the server. Instead, it returns a `Value::NativeObject` representing the server config. A new `Http.serve(app, port)` builtin in the interpreter (not the plugin) actually runs the loop, because it has `&mut self` access to call closures.

Revised approach:
- FFI plugin functions handle **data operations** (create app, add route, parse request)
- The **serve loop** is a special interpreter builtin that has closure-calling capability
- This sidesteps the callback problem entirely

### Task 6: Implement HTTP serve loop in interpreter

**Files:**
- Modify: `crates/opal-interp/src/eval.rs`

**Add a special method call handler:** When `app.run!(port)` is called on a NativeObject (the router), the interpreter:

1. Gets the router's routes from the native object
2. Binds `TcpListener` to the port
3. Accept loop:
   a. Read raw bytes from stream
   b. Parse HTTP request line (`GET /path HTTP/1.1`)
   c. Extract path params (`:name` segments)
   d. Match against routes
   e. Call the matching Opal closure with params as a dict (or a simple object)
   f. Write HTTP response

## Key Risk: Handler Callback

The biggest technical risk is calling Opal closures from inside the HTTP serve loop. The solution: keep the serve loop inside the interpreter (not in the FFI plugin), so it has `&mut self` access to `call_closure`. The plugin only handles data — creating the router, adding routes, extracting route info. The interpreter handles execution — the TCP loop, request parsing, and closure invocation.
